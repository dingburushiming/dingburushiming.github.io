服务注册中心Zookeeper

Zookeeper是Apache Hadoop的子项目，是一个树型的目录服务，支持变更推送，适合作为Dubbo服务的注册中心

流程说明

  • 服务器提供者Provider启动时，向/dubbo/com.foo.BarService/providers目录下写入自己的URL地址
  • 服务消费者Consumer启动时，订阅/dubbo/com.foo.BarService/providers目录下的提供者URL地址。并向/dubbo/com.foo.BarService/consumers目录下写入自己的URL地址
  • 监控中心Monitor启动时，订阅/dubbo/com.foo.BarService目录下的所有提供者和消费者URL地址

安装zookeeper

  1. 下载3.4.6版本的zookeeper http://archive.apache.org/dist/zookeeper/zookeeper-3.4.6/
  2. 将文件夹放到Linux的/usr/local目录下
  3. 在zookeeper-3.4.6目录下，创建data目录
  4. 进入conf目录，把zoo_sample.cfg改名为zoo.cfg
  5. 打开zoo.cfg文件，修改data属性：dataDir=/usr/local/zookeeper-3.4.6/data
  6. 使用chmod 777 zkServer.sh给系统读写的可执行权限
  7. ./zkServer.sh start启动zookeeper
  8. ./zkServer.sh status查看是否启动成功

一个简单的dubbo案列

总体流程

监控

主要是监测zookeeper，进而监测所有服务提供方和消费方

修改application.properties配置文件，运行dubbo-admin项目，通过1270.0.1:7001(本地启动)

公共接口抽离

将微服务中共用的接口，类抽离到一个新的项目中，各个微服务项目通过maven依赖获取公共数据

服务提供方

所有的微服务(提供方，服务方)都需要引入公共接口，dubbo,zookeeper的maven依赖

服务消费方

配置文件的命名空间非常重要，相同与导包，不能写错

配置文件的覆盖策略

虚拟机参数>xml配置文件>properties文件

方法级>接口级>全局配置，越具体的优先级越高

消费方>提供方

级别一样，则消费者优先，提供方次之，方法级别不同则按照级别，方法级别优先级高于服务消费/提供方

SpringBoot整合dubbo

方式一：基于注解

导入dubbo-starter,在application.properties配置属性，使用@Service暴露服务，使用@Reference引入服务

注解 作用 位置
@Service 暴露注解 标注在service方法上，将该方法暴露出来供其他服务远程调用
@EnableDubbo 开启基于注解的dubbo 标注在SpringBoot启动类上
@Reference 替代@Autowired，远程调用类对象 标注在bean上

基于注解的application.properties全局配置文件

方式二：基于xml文件

保留dubbo的xml文件，在启动类上标注@ImportResource(locations=”classpath:provider.xml”)引入相关配置

provider.xml配置文件

方法三：基于配置类

将每一个组件手动创建到容器中，在启动类上标注@EnableDubbo(scanBasePackages=”com.atguigu.gmall”)设置扫描范围，范围内的配置文件才会生效

高可用

Zookeeper宕机

Zookeeper只是一个数据记录中心，如果宕机，本地服务如果有缓存会记录目标服务的位置；也可以在调用时直接指定目标服务，采用直连的方式跳过Zookeeper，即调用服务时使用@Reference(url = “127.0.0.1:20882”)，直接获取目标服务。

负载均衡

基于权重的随机负载均衡机制(Random LoadBalance)

基于权重的轮询负载均衡机制(RoundRobin LoadBalance)

最少活跃数-负载均衡机制(LeastActive LoadBalance)

一致性hash-负载均衡机制(ConsistentHash LoadBalance)

使用方法

在@Reference中设置被调用类型，random，roundrobin，leastactive

服务降级

服务降级是针对消费方的服务

服务器压力剧增的情况下，根据具体情况，对一些服务和页面有策略的不处理或换种简单的方式处理，从而释放服务器资源以保证核心交易正常运行或高效运作

不调用方法直接返回null,mock=force:return+null

消费方对该服务方法的调用直接返回null值，不发起远程调用，用来屏蔽不重要服务不可用时对调用方的影响

服务容错

调用失败后返回null,mock=fail:return+null

消费方调用服务方法的失败后，再返回null值，用来容忍不重要服务不稳定时对调用方的影响

Hystrix

导入Hystrix启动器依赖

配置启动类注解

在方法上配置异常时处理容错的注解，没错正常运行，有错报错或者执行回调方法

RPC&Netty原理

一次完整的同步RPC调用流程

  1. 服务消费方client调用以本地调用方式调用该服务

  2. client stub接收到调用后负责将方法，参数等组装成能够进行网络传输的消息体

  3. client stub找到服务地址，并将消息发送到服务端

  4. server stub接收到消息后进行解码

  5. server stub根据解码结果调用本地的服务

  6. 本地服务执行并将结果返回给server stub

  7. server stub将返回结果打包成消息并发送至消费方

  8. client stub接收到消息，并进行解码

  9. 服务消费方得到最终结果

    底层网路通讯是基于框架Netty,而Netty是基于Java的NIO(Non-Blocking IO,非阻塞IO )

各种IO的区别

IO( Input/Output)阻塞IO

BIO(Blocking IO)阻塞IO

NIO(Non-Blocking IO)非阻塞IO

Selector 一般称 为选择器，也可以翻译为 多路复用器

Connect（连接就绪）,Accept（接受就绪）,Read（读就绪）,Write（写就绪）
